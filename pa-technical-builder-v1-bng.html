    Introduction

    What this document covers · and who it's for.

    Purpose. The scope of this document is to describe the solution for the AI Voice Personal Assistant service for Operator — a turn-key intercept layer that answers calls the subscriber can't take, captures intent, and turns dead air into engagement.

    Audience. This document is intended for all members involved in the implementation of the AI Personal Assistant service at Operator — Marketing, CORE/Transmission, Network, IT/Infrastructure, Revenue Assurance, and product teams.

    Why now, why Operator

    Mobile markets with high voice-call density and tech-receptive subscriber bases see measurable revenue and engagement gaps from missed calls. The PA platform plugs directly into Operator's network — no handset changes, no app dependency — delivering value from day one of provisioning.

    Platform Overview

    Three call-termination scenarios. One AI intercept layer.

    PA is provisioned at the HLR by setting three call-forwarding flags so that any unanswered call routes to the PA platform instead of dropping or going to voicemail. The flags map directly to the three GSM call-termination conditions:

    CFNRc · UNREACHABLE

    Switched Off / No Coverage

    Subscriber's handset is powered off, in flight mode, or out of network coverage. Call never even rings the device.

    → PA answers, captures caller's message
    CFB · BUSY

    Busy / Rejected

    Subscriber is on another call, or actively rejects the incoming call. Network responds with 486 Busy.

    → PA answers, plays personalised greeting
    CFNRy · NO REPLY

    No Answer / Timeout

    Phone rings but subscriber doesn't pick up within the operator-configured ring-timeout (typically 20–30s).

    → PA answers, AI engages caller

    HLR Flag Strategy

    Provisioning is what makes the call land on PA.

    PA does not validate every call. The HLR is the source of truth: forwarding rules are reconciled at provisioning time, and on every inbound call the MSC consults HLR and routes the call to either the PA platform (active subscriber) or directly to the operator default (non-subscriber). PA never sees a non-subscriber call.

    FlagGSM ConditionSet To
    CFBCall Forwarding on BusyPA shortcode (when subscriber Active)
    CFNRyCall Forwarding on No ReplyPA shortcode (when subscriber Active)
    CFNRcCall Forwarding on Not ReachablePA shortcode (when subscriber Active)
    Recommended provisioning path: via Mediator. The PA platform calls a provisioning team / mediation layer (SOAP, XML/HTTP, or MAP) which writes to HLR on PA's behalf. Direct PA → HLR connectivity is also supported where security policy permits. Functionally equivalent — differs only in network path.
    Call Routing

    What happens on a real call.

    3.1 · B-party subscribed to PA
    HLR holds CFB / CFNRy / CFNRc → PA. When B is busy / no-reply / unreachable, MSC forwards the INVITE to PA. PA performs a defensive subscription check (always returns Active), answers, captures the message, pushes notification to B over WhatsApp / SMS / app.
    B-party subscribed — PA handles the call

    3.2 · B-party not subscribed

    Routing for non-PA subscribers depends on which operator default is configured on HLR. Priority: MCA > VM > greenfield. Only the relevant box shows below.

    HLR holds no special forwarding rule (greenfield). Call drops normally — busy/disconnect tone returned to A. PA is not invoked. No per-call validation, no SIP REFER fallback.
    HLR holds the operator default (existing MCA service). MSC routes the unanswered call directly to MCA without invoking PA. This is the standard operator missed-call experience — unchanged. If VM is also active, MCA takes precedence at the HLR level for the unanswered-call flags.
    HLR holds VM as the operator default. MSC routes to VM. PA is not invoked.

    Subscription Lifecycle

    Three states. One state machine.

    Every PA subscriber is in one of three states. Transitions are driven by events (subscribe, charge success, charge failure, unsubscribe, grace expiry).

    Why Grace exists. Without a Grace state, every prepaid charge failure (low balance at renewal moment) would silently drop the subscriber to Inactive. With Grace, PA is temporarily disabled at HLR (calls go to operator default) but the user is recoverable: PA Server retries the charge during the configurable window. On a successful top-up the user returns to Active without re-onboarding. This separates "temporarily can't charge" from "has unsubscribed" — protecting subscribers from accidental hard-churn.
    System Architecture

    Where PA runs. What sits around it.

    The AI Personal Assistant is built around a robust, scalable, modular architecture. At its core, the Brain integrates advanced AI capabilities with a comprehensive suite of core components, external services, and interaction channels.

    7.1 · Network architecture · deployment view

    The PA platform is deployed inside Operator's premises, behind the operator firewall. Outbound access to cloud TTS / STT / LLM platforms is allow-listed at the firewall. A site-to-site VPN connects the deployment to blackNGreen operations for remote OAM only.

    Fig. 7.1 · Deployment Architecture · Cloud platforms ↔ Operator firewall ↔ Operator premises ↔ blackNGreen NOC

    • The system integrates with cloud platforms for LLM, STT, and TTS to enhance AI-driven interactions, while keeping all subscriber data inside the operator network.
    • A secure operator firewall protects access to the network, ensuring controlled communication between external cloud services and internal systems.
    • The operator network connects to key nodes — HLR, MSC, IN/OCS, SDP, SMSC, USSD Gateway — for handling call routing, charging, and messaging.
    • The IVR platform consisting of Application VMs and a DB Cluster manages voice interactions and stores relevant data inside operator premises.
    • A site-to-site VPN connection allows remote OAM access from the blackNGreen operations centre for monitoring and configuration only.

    Smart Promo · Conversion Engine

    Every missed call is a moment. Captured, at scale.

    PA's biggest growth lever isn't the campaign — it's the moment. The Smart Promo Engine targets users at the exact second they experience the loss of a call, then offers them the cure. Two methods, one rotation engine, fully configurable by the operator.

    The opportunity, in one number

    5–12% Subscription rate observed in high-value segments at the second exposure · zero acquisition CAC

    Unsubscribe & State Restoration

    Three triggers, one symmetric flow.

    Unsubscribe handling is symmetric across three trigger types — user-initiated, subscription expiry, and grace-window expiry. All three converge on the same HLR rule: erase PA forwarding flags and restore them to the operator default.

    TriggerSourceHLR Outcome
    User-initiatedIVR "unsubscribe" / SMS "STOP" / USSD *XXX*0#Erase PA flags · restore to default
    Subscription expirySystem-triggered at end of pack (no auto-renew or charge fail beyond grace)Erase PA flags · restore to default
    Grace exitSystem-triggered when grace window closes without successful chargeErase PA flags · restore to default

    Fallback Path · Option B

    USSD self-service. The least-recommended option.

    If the operator's HLR provisioning interface is not yet available at PA launch — for instance during a quick pilot, or while the SOAP/XML mediator is being wired up — there is a fallback path: the subscriber dials standard GSM CFB / CFNRy / CFNRc USSD codes themselves to forward calls to PA. Every call lands on PA, which then performs per-call subscription validation and releases non-subscribers via SIP REFER.

    Why we don't recommend this. Two operational drawbacks: (1) the user-subscription experience is hindered by the manual USSD dialling step (high drop-off), (2) per-call SIP REFER is heavier on the signalling layer than HLR-flagging. Use as a transitional path only — switch to the recommended HLR provisioning flow as soon as it's available.

    User-side activation

    Instead of PA setting HLR flags via mediator, the subscriber sets them on their own handset using GSM standard codes:

    When the operator's HLR provisioning interface becomes available, switch back to the recommended flow (§5 HLR Flag Strategy) for both user experience and signalling efficiency. PA's subscription DB and call-handling logic stay the same — only the provisioning path changes.

    Data Security

    How subscriber data is protected.

    In the system, ensuring the security of sensitive and Personally Identifiable Information (PII) data is paramount. The implementation of data encryption both at rest and in motion is undertaken meticulously.

    Data at Rest

    • The data stored within the Elasticsearch cluster and MySQL databases is encrypted using Advanced Encryption Standard (AES) with a 256-bit key size. Applied at the file-system level using eCryptfs or EncFS.
    • Database encryption is conducted through MySQL's built-in AES encryption functions for specific data fields containing sensitive information.

    Data in Motion

    • SSL / TLS with AES 256-bit encryption protects data as it is transmitted to and from the Elasticsearch cluster and MySQL databases.
    • API Security — All APIs are secured with TLS, and OAuth 2.0 is used for authorisation, ensuring only authenticated services can access data.

    Sensitive Data Protected

    • Fields containing PII — such as customer identification numbers and call-summary information — are selectively encrypted using MySQL's AES_ENCRYPT() function.
    • Elasticsearch's built-in HTTPS support is configured to use AES 256-bit encryption for data in transit.
    Key management & auditing. Access to the encryption keys is strictly controlled and monitored, and the keys themselves are rotated periodically to mitigate the risk of unauthorised key usage. Data access patterns are logged to detect any unauthorised attempts. Regular audits ensure compliance with security policies and standards, and incident-response plans are established to address potential data breaches.

    In the Operator deployment the entire platform runs on operator-side infrastructure inside Operator premises (see Integration Points). External exposure is therefore minimal — no PA component is reachable from the public internet, and outbound traffic is restricted to allow-listed LLM / TTS endpoints through the operator's firewall.

    Data Moderation

    How hallucinations are prevented.

    At blackNGreen, we understand that accuracy and reliability are critical when deploying AI-powered solutions. To ensure the assistant provides precise, contextually relevant, and trustworthy responses while avoiding hallucinations (incorrect or fabricated information), we have implemented multiple guard rails.

    Dynamic Knowledgebase Access

    Unlike generic AI models that generate text from probabilities, the assistant retrieves information from pre-verified enterprise knowledgebases. This ensures that responses are based on official Operator data, reducing the risk of hallucinations.

    AI Moderation & Compliance Filters

    The assistant incorporates real-time AI moderation to filter responses for accuracy, policy adherence, and regulatory compliance. The moderation system ensures that:

    • The assistant does not generate speculative or misleading content.
    • Responses align with Operator guidelines and industry standards.
    • Any unverified or ambiguous queries are escalated to human agents instead of generating uncertain answers.

    17.2 · Sizing formula & assumptions

    Concurrent calls — the primary driver of signalling-VM count — are computed as follows:

    Capacity formula
    • Concurrent calls = BHCA × MHT / 3600
    • BHCA = Busy Hour Call Attempts — the peak hour PA-flagged inbound volume.
    • MHT = Mean Holding Time per PA call. Typical MHT is 25–45 seconds (greeting + capture + hangup; OBD calls trend slightly longer).
    • N+1 redundancy on signalling VMs — one extra node above peak demand, so a node loss doesn't degrade service.
    • LLM / TTS calls are async — they reach cloud providers (Azure, Gemini, Google Cloud) over allow-listed FQDNs and do not block signalling capacity.

    • Inside Operator premises — all VMs sit on the operator network, behind the operator firewall.

    FAQ · Coexistence

    The coexistence questions operators always ask.

    PA never displaces a paid service silently, and PA always restores the prior default on opt-out.

    Subscriber already has a free MCA service?
    Take the flag, give it to PA. PA does MCA by default as a standard built-in feature — the subscriber gets a richer summary instead of a plain "you have a missed call" SMS. No notification needed, no service is lost.
    Subscriber has a paid MCA or voicemail subscription?
    Two options at deployment time:
    • SMS the user — "Your MCA / VM is still active. Reply STOP to avoid double-billing."
    • Auto-deactivate via SDP — Operator (or blackNGreen, if managing flagging) clears the prior subscription on PA activation. No double-charge.
    Voicemail is the existing default?
    PA flags override VM. Strict UX upgrade — text summary instead of a callback-and-listen flow. On unsubscribe, VM is restored. VM users are a natural VM-Plus upsell via the Smart Promo Engine.
    Why a long code, not a short code?
    Long codes go through standard carrier billing. Operator earns inter-leg revenue on every captured call — more PA traffic means more billable minutes. No revenue gap.
    Charging fails repeatedly — hard-churn?
    No. The subscriber enters Grace (see Lifecycle). PA disabled, calls fall back to operator default, balance is retried during the window. On top-up they return to Active without re-onboarding. Only on Grace expiry do they drop to Inactive — and the prior default is fully restored.
